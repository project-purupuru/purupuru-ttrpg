## What's an indexer? (skip if you've built one before)

A Solana program emits **structured logs** when its instructions run successfully. These logs sit in the validator's transaction log forever, but they're not queryable in any useful way without a process that:

1. **Subscribes** to the program's logs in real-time (WebSocket from a Solana RPC node)
2. **Parses** the base64-encoded event data using the program's IDL (interface definition file generated by Anchor)
3. **Writes** structured rows to a database (Postgres in our case · same one Score already uses)
4. **Exposes** queries to the frontend (Hasura / GraphQL in our case · already wired up)

The pattern is exactly the same as Ethereum log indexers (Subgraph · Goldsky etc.) — just running over Solana's RPC. The TS-side ecosystem is mature: `@solana/web3.js` for RPC · `@coral-xyz/anchor` for IDL parsing · pino for logging.

### Two architecture choices

You'll pick one of these two shapes — they have different trade-offs:

| Shape | What | Pros | Cons |
|---|---|---|---|
| **Tail mode** | Subscribe to `connection.onLogs(programId)` · live | real-time · simple · no backfill | misses events when process is down · no historical sweep |
| **Sweep + tail** | Periodic `getSignaturesForAddress(programId)` cursor + onLogs for live | resilient · catches up after downtime · backfills demo prep | more code · cursor management · more RPC calls |

For hackathon scope: **tail mode** is fine. zksoju will trigger a fresh batch of devnet `StoneClaimed` events the morning of demo recording so the indexer has fresh data to display, and the process needs to stay up only for the demo window. Sweep+tail is the post-hackathon hardening shape.

### Why this is non-obvious

Solana doesn't have a built-in event log like Ethereum's `event` keyword. Anchor adds events via **structured logs**: when `emit!(StoneClaimed { ... })` runs in our program, Anchor base64-encodes the struct (including a discriminator hash) and writes it as a `Program log:` line in the transaction log.

Your indexer reads those `Program log:` lines, finds the ones starting with the `StoneClaimed` discriminator, base64-decodes the body, and parses the struct using the IDL.

The IDL (JSON file) is the contract. Anchor generates it at build time. We've published it · path below.

## The event you're indexing · `StoneClaimed`

When a user successfully claims a genesis stone, our program emits this event. This is the ONLY event our program emits in v0:

```rust
#[event]
pub struct StoneClaimed {
    pub wallet: Pubkey,    // the user wallet that claimed
    pub element: u8,       // 1=Wood · 2=Fire · 3=Earth · 4=Metal · 5=Water
    pub weather: u8,       // cosmic weather byte at claim time (1-5 same encoding)
    pub mint: Pubkey,      // the new NFT mint that was created
}
```

After Anchor's base64 decode, you'll get this shape (assuming TS):

```ts
type StoneClaimed = {
  wallet: PublicKey
  element: 1 | 2 | 3 | 4 | 5
  weather: 1 | 2 | 3 | 4 | 5
  mint: PublicKey
  // metadata you'll want to add yourself:
  signature: string       // tx signature where this event was emitted
  slot: number            // Solana slot · for ordering
  blockTime: number | null  // unix timestamp · null until block finalized
}
```

### Element / weather byte encoding

Both fields use the same 1-5 mapping (the wuxing five-element cycle):

| Byte | Element | Notes |
|---|---|---|
| 1 | Wood | growth · spring |
| 2 | Fire | momentum · summer |
| 3 | Earth | grounding · late-summer |
| 4 | Metal | clarity · autumn |
| 5 | Water | flow · winter |

`element` = the user's archetype (computed from quiz answers · server-validated).
`weather` = the cosmic weather imprint at the moment of mint (currently a deterministic stub keyed off day-of-week · sprint-3 wires real oracle if time permits).

For the dashboard, you'll likely want to surface both — the awareness-layer story is "elements of users + elements of cosmic weather, side by side."

### Suggested data model

Postgres table sketch (your call · this is just a starting point):

```sql
CREATE TABLE purupuru_stone_claims (
  signature   TEXT NOT NULL,           -- Solana tx signature
  log_index   INT NOT NULL,            -- which log within the tx
  slot        BIGINT NOT NULL,         -- ordering key
  block_time  TIMESTAMPTZ,             -- nullable until finalized
  wallet      TEXT NOT NULL,           -- base58 pubkey
  element     SMALLINT NOT NULL,       -- 1-5
  weather     SMALLINT NOT NULL,       -- 1-5
  mint        TEXT NOT NULL,           -- base58 pubkey
  PRIMARY KEY (signature, log_index)
);

CREATE INDEX idx_stone_claims_slot ON purupuru_stone_claims (slot DESC);
CREATE INDEX idx_stone_claims_element ON purupuru_stone_claims (element);
```

The compound primary key `(signature, log_index)` makes the indexer **idempotent on retry** — if the indexer process restarts and re-reads recent slots, it can `INSERT ... ON CONFLICT DO NOTHING` without polluting.

## Reference implementation pattern (TypeScript · the path zksoju recommends if you go from zero)

```ts
import { Connection, PublicKey } from "@solana/web3.js"
import { BorshCoder, EventParser } from "@coral-xyz/anchor"
import idl from "./purupuru_anchor.json"

const PROGRAM_ID = new PublicKey("7u27WmTz2hZHvvhL89XcSCY3eFhxEfHjUN5MjzMY6v38")
const RPC = "https://api.devnet.solana.com"
const WS  = "wss://api.devnet.solana.com"

const connection = new Connection(RPC, { wsEndpoint: WS, commitment: "confirmed" })
const coder = new BorshCoder(idl as any)
const eventParser = new EventParser(PROGRAM_ID, coder)

// Subscribe · runs forever
connection.onLogs(
  PROGRAM_ID,
  async (logs, ctx) => {
    if (logs.err) return
    const events = eventParser.parseLogs(logs.logs)
    for (const event of events) {
      if (event.name === "StoneClaimed") {
        const data = event.data as {
          wallet: PublicKey
          element: number
          weather: number
          mint: PublicKey
        }
        await persistStoneClaim({
          signature: logs.signature,
          slot: ctx.slot,
          wallet: data.wallet.toBase58(),
          element: data.element,
          weather: data.weather,
          mint: data.mint.toBase58(),
        })
      }
    }
  },
  "confirmed",
)
```

Total LOC for a working v0 indexer: ~50-100 lines. Most of the time is in:

1. Setting up the Postgres connection
2. Wiring the persistence layer + Hasura exposure
3. Building the dashboard UI on top

The Solana subscription itself is trivial. The two non-obvious bits:

1. **`commitment: "confirmed"`** vs `"finalized"` — confirmed = ~5s latency · finalized = ~30s but unrevertable. For demo, confirmed is fine.
2. **Gracefully handling RPC disconnects** — `onLogs` will silently stop receiving events if the WebSocket dies. Wrap with a health check / reconnect loop.
